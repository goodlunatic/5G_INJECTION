20 MHz retest summary after log cleanup and RF timing/gain adjustments (2026-04-07)

Cleanup performed
- Deleted older session_logs directories.
- Cleared testbed/logs before rerun.

Configuration used in final retest
- X310 gNB: clock=external, sync=internal, time_source=internal
- B210 UE: clock=external, time_source=internal
- gNB gains: tx_gain=31.5, rx_gain=31.5
- UE gains: tx_gain=80, rx_gain=20
- Bandwidth: 20 MHz

What was improved
- gNB now uses a valid X310/UBX gain range instead of the previous out-of-range tx/rx defaults.
- gNB attaches to AMF successfully.
- UE consistently finds the cell and decodes MIB/SIB1 successfully.
- gNB consistently detects UE PRACH and schedules RAR.

Current blocker
- UE still does not complete random access / RRC setup.
- gNB receives the expected Msg3/PUSCH after RAR scheduling, but every attempt fails CRC.
- Observed PUSCH SINR remains negative (about -11.6 dB to -27.0 dB in this run).
- Because Msg3 is not decoded, NGAP InitialUEMessage / NAS Registration Request never reaches Open5GS.

Core-network observation
- Open5GS receives gNB-N2 connection only.
- No UE-side registration/authentication/PDU-session signaling was observed in Open5GS logs in this run.

Interpretation
- The remaining problem is on the over-the-air uplink path between UE and gNB, not on the core network.
- This now looks like an RF/link-budget or uplink impairment issue (for example attenuation/gain/antenna path/duplex isolation/cabling), not a clock-lock or AMF startup issue.
